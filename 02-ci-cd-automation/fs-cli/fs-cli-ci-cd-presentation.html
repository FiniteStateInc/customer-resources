A field guide · v2.0.x
Where it fits

What fs-cli does

  • Detects package managers across 19 ecosystems — npm, Go, Maven, pip, Cargo, and more.
  • Resolves the full dependency tree with scope, emits a CycloneDX SBOM.
  • Uploads to the Finite State platform — analysis runs platform-side.
  • Async by design: the scan submits and returns; it never blocks CI agents on analysis.
choose the right command

scan vs. upload

scan → source repos with package managers. Resolves the dependency tree → SBOM.

upload → a pre-built binary artifact for analysis.

SBOMs and external scanner output have their own commands: import · third-party.

fs-cli owns step 3 — scan and upload. Reporting, gating, and PR feedback layer on top via the Finite State Actions suite (or fs-report). download-sbom can export the result at the end.

Step 02

Installing in CI

  • Publicly available — install freely from anywhere, no auth needed to fetch the binary.
  • The install script is the quickest path in CI; go install or build from source work too.
  • Pin a version for reproducibility; pass --no-update-check to skip the self-update probe.
  • Cache the binary across runs. Linux / macOS / Windows runners all supported.
.github/workflows/finite-state.yml
# install fs-cli
- name: Install fs-cli
  run: |
    curl -fsSL \
      https://raw.githubusercontent.com/FiniteStateInc/\
      customer-resources/main/02-ci-cd-automation/\
      fs-cli/install.sh | sh

Step 03

Running a scan

  • fs-cli scan . → detect → scan → assemble → upload, concurrent under the hood.
  • Detection priority resolves overlaps: Maven > Gradle, lock file > package.json.
  • Exec scanners (Maven, Gradle, SBT, Go) need the build tool on PATH — bake it into the CI image.
  • --scope runtime drops dev & test deps from the inventory.
scan step
fs-cli scan . \
  --token "$FS_API_TOKEN" \
  --endpoint "https://$FS_DOMAIN" \
  --name "$REPO_NAME" \
  --project-id "$FS_PROJECT_ID" \
  --version "$REF-$SHA"
Don't skip this

Versioning strategy

Every scan needs a unique version. Reuse one and you overwrite the previous scan — always fold in the git SHA.

TriggerPatternExample
Tag / release<tag>-<sha>v1.4.2-9af3c1
Pull requestpr-<number>-<sha>pr-318-9af3c1
Schedulednightly-<date>-<sha>nightly-20260624-9af3c1

Branch, commit, and remote git metadata are captured automatically and attached to the version.

The reporting half

fs-report — inventory in, findings out

  • fs-cli writes the inventory up; the platform analyzes it; fs-report reads the findings back.
  • A separate Python CLI — pipx install fs-report — the findings & reporting engine.
  • Runs recipes that turn findings into deliverables: HTML · CSV · XLSX · JSON · MD · PDF.
  • The GitHub run-report action simply wraps it.

Setup

Installing fs-report

  • pipx install fs-report — it's a Python tool; pin a version in CI.
  • Auth via env: FINITE_STATE_AUTH_TOKEN + FINITE_STATE_DOMAIN — the long names, not fs-cli's FS_*.
  • For PDF output, pre-stage Chromium with install-engine — otherwise it pulls ~150 MB on first render.
  • Optional AI: set ANTHROPIC_API_KEY (or OpenAI / Gemini / GITHUB_TOKEN) to enable --ai.
finite-state.yml — install steps
env:                  # job-level, inherited by all steps
  FINITE_STATE_AUTH_TOKEN: ${{ secrets.FS_API_TOKEN }}
  FINITE_STATE_DOMAIN:     ${{ vars.FS_DOMAIN }}
steps:
  - run: pipx install fs-report
  - run: fs-report install-engine --with-deps  # PDFs only
In the pipeline

Running fs-report headless

  • --headless → non-interactive, and it prints a JSON summary to stdout you can parse to gate.
  • Scope it: --project / --folder / --version; pick work with repeated --recipe.
  • --cache-ttl 1h keeps reruns fast; --output ./reports, then upload as an artifact.
  • Push triage back: --autotriage / --apply-vex-triage write VEX to the platform.
fs-report run
fs-report run \
  --recipe "Triage Prioritization" \
  --recipe "Version Comparison" \
  --project openwrt --period 30d \
  --headless --cache-ttl 1h --output ./reports

# stdout →
{ "success": true,
  "recipes": [ { "recipe": "Triage Prioritization",
                 "stats": { "total_findings": 142 } } ] }
Layer on top

Gating & reporting

  • fs-cli uploads the inventory; gating & reporting layer above it.
  • Chain: setup → scan → run-report → quality-gate → pr-comment.
  • pr-comment edits in place — use if: always() so it posts even on failure.
  • download-sbom exports the SBOM (with VEX) as a build artifact.
gate modes — AND'd

Three ways to fail a build

delta threshold triage-priority

e.g. max-new-critical: 0 + fail-on-p0: true.

✓ pass → merge ✕ fail → block
fs-cli's built-in gate

Gate natively with query

The exit code is the gate. A read-only platform query that exits non-zero to fail the build — no fs-report, no report files to parse.

--type scan

Did scanning finish?

Checks completion across every scan type — SCA, config, binary SAST, reachability. --wait polls until settled; fails if any scan failed (or, with --fail-on-scan-incomplete, is still running).

--type project

Per-finding AND gate

Prints severity counts, then fails if one finding matches every condition you set. Always fails if the version was never successfully scanned — no false green.

native gate in CI
# fail on a reachable, KEV-listed critical
fs-cli query --type project \
  --name "$REPO" --version "$REF-$SHA" \
  --fail-on-severity critical \
  --reachable --vulns-in-kev
conditions · AND-combined
--fail-on-severity --reachable --vulns-in-kev --vulns-in-vc-kev --exploit-maturity --max-epss

A lightweight pass/fail — reach for the fs-report + quality-gate chain when you also want reports and PR comments. --format json for machine output; distinct exit codes per failure mode are planned.

Worked example

Recipe: the PR gate

.github/workflows/finite-state.yml
on: { pull_request: { branches: [main] } }
jobs:
  security:
    runs-on: ubuntu-latest
    permissions: { contents: read, pull-requests: write }
    steps:
      - uses: actions/checkout@v4
      - run: curl -fsSL .../fs-cli/install.sh | sh
      - run: fs-cli scan . --name "$REPO" --project-id "$PID"
              --version "pr-${{ github.event.number }}-${{ github.sha }}"
      - uses: FiniteStateInc/finite-state-actions/actions/run-report@v1
        with: { recipe: "Triage Prioritization,Version Comparison" }
      - uses: .../quality-gate@v1
        with: { mode: delta,triage-priority, max-new-critical: 0, fail-on-p0: true }
      - uses: .../pr-comment@v1
        if: always()
        with: { template: triage }
When it goes sideways

Operations & troubleshooting

common failures

Read the error

401 / 403 → token invalid or under-scoped.
timeout → raise the wait window for large scans.
tool not found → exec scanner missing from the image.
file not found → build step ran after the scan.

diagnostics

Turn up the volume

--debug for structured logs, --quiet to silence.

Local-env gotcha: stray FS_* vars in your shell can change behavior — unset them when reproducing CI locally.

Exit code 0 = pass, non-zero = fail — that's the contract CI reads to block or merge.

Take it home

Three things to remember

01

Scan, don't block

fs-cli submits async — your pipeline keeps moving.

02

Version every scan

Always fold in the SHA, or you overwrite history.

03

Gate where it counts

delta + triage-priority blocks the PRs that matter.

Pick a template, drop in three secrets, and you have security gating on every pull request.
