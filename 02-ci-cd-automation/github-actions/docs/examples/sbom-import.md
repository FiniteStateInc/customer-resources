# Example: SBOM Import

Import SBOMs generated by external build systems (Yocto, Buildroot, etc.) into Finite State for vulnerability analysis and triage.

## When to use this

- Your build system already generates SBOMs (CycloneDX or SPDX)
- You want to analyze the SBOM in Finite State rather than uploading the raw binary
- Your binary is too large or impractical to upload directly
- You use a multi-stage build where SBOM generation happens in an earlier stage

## Workflow

```yaml
name: SBOM Import & Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read
  pull-requests: write

jobs:
  analyze:
    name: Import & Analyze SBOM
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Your build step that generates the SBOM.
      # Replace this with your actual build process.
      - name: Build and generate SBOM
        run: |
          make build
          make sbom  # outputs build/sbom.cdx.json

      # Authenticate with Finite State
      - uses: finite-state/setup@v1
        with:
          api-token: ${{ secrets.FS_API_TOKEN }}
          domain: ${{ vars.FS_DOMAIN }}
          project-id: ${{ vars.FS_PROJECT_ID }}

      # Import the CycloneDX SBOM
      - uses: finite-state/upload-scan@v1
        id: upload
        with:
          type: sbom
          sbom-format: cdx
          file: build/sbom.cdx.json
          version: "pr-${{ github.event.number || github.ref_name }}"

      # Run reports on the imported SBOM
      - uses: finite-state/run-report@v1
        id: report
        with:
          recipe: "Triage Prioritization"
          period: 30d

      # Post findings summary
      - uses: finite-state/pr-comment@v1
        if: github.event_name == 'pull_request' && always()
        with:
          template: summary
          report-dir: ${{ steps.report.outputs.report-dir }}
```

## SPDX variant

To import an SPDX SBOM instead of CycloneDX:

```yaml
- uses: finite-state/upload-scan@v1
  with:
    type: sbom
    sbom-format: spdx
    file: build/sbom.spdx.json
    version: "pr-${{ github.event.number }}"
```

## Yocto integration

For Yocto builds that generate SBOMs via `meta-spdxscanner` or similar:

```yaml
- name: Build Yocto image
  run: |
    source oe-init-build-env
    bitbake core-image-minimal

- uses: finite-state/upload-scan@v1
  with:
    type: sbom
    sbom-format: spdx
    file: build/tmp/deploy/spdx/core-image-minimal.spdx.json
```

## Key Points

1. **SBOM format:** Use `cdx` for CycloneDX (`.cdx.json`) or `spdx` for SPDX (`.spdx.json`). Finite State accepts both.

2. **Analysis depth:** SBOM imports get full vulnerability matching and triage scoring, but reachability analysis requires the actual binary. If you need reachability data, upload the binary via `type: sca` instead.

3. **Version tracking:** Use meaningful version names so you can track changes over time in the Finite State platform.
